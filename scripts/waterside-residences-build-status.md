## Findings

### What the brief assumed

- A self-contained `Downtown66Page.tsx` exists at `src/pages/properties/`
- New listings are added by creating a sibling `WatersideResidencesPage.tsx`
- R2 image URLs are inlined as TSX string constants at the top of the listing TSX

### What the repo actually does

1. **Listings are JSON, not TSX.** Each property is three data files:
   - `data/properties/{slug}-source.json` — author-written, validated against `propertySourceSchema` (Zod) in `src/components/property/schema.ts`
   - `data/properties/{slug}-images.json` — generated by `scripts/import-property-images.ts`, contains ImageVariants (hero/gallery/thumb URLs on R2) for each image key
   - `data/properties/{slug}.json` — generated by `scripts/scaffold-property.ts`, the merged listing with computed JSON-LD, SEO, and baked image URLs

2. **Auto-discovery.** `src/data/propertyListings.ts` uses `import.meta.glob` to load every `data/properties/*.json` (skipping `-source.json` and `-images.json`). Drop a valid `{slug}.json` in and the route picks it up.

3. **Routing.** A single generic `src/pages/PropertyDetail.tsx` reads the slug from the URL, looks up the rich listing, and renders via `src/components/property/PropertyListingPage.tsx`.

4. **Image pipeline.** `scripts/import-property-images.ts` uses `@aws-sdk/client-s3` against the R2 endpoint with credentials from `.env.local`. It generates three variants per image (hero 2000px q85, gallery 1200px q82, thumb 400px q80) using sharp and uploads them to R2.

5. **Section components are fixed and schema-bound.** `PropertyListingPage` composes from a closed set of sections in `src/components/property/sections/`: Hero, AtAGlance, Pricing, FloorPlans, Gallery, Amenities, Walkthrough, LocationMap, Building, Connectivity, Builder, Neighbourhood, InvestmentThesis, PaymentPlan, EmiCalculator, ContactBlock, Faq, Rera, Disclaimer, SimilarProperties. You do not add a section by writing TSX inside a property file.

### Brief instructions that cannot run as written

| # | Brief instruction | Why it fails |
|---|---|---|
| 1 | "Self-contained single .tsx file for the listing page" | Architecture is data-driven JSON. There is no precedent for a per-property TSX. |
| 2 | "All R2 assets referenced as inline URL string constants at the top of the TSX file" | R2 URLs live in `{slug}-images.json` as ImageVariants. The schema requires image keys, not raw URLs. |
| 3 | "Hotlink amenity icons, floor plans, master plan, secondary gallery from krisumi.com" | The `floorPlans` schema requires `imageKey` mapping to the images manifest (R2).
